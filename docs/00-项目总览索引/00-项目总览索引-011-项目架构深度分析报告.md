## 二、关键问题诊断

### 2.1 🔴 P0 — Supabase 强耦合（核心阻塞）

**问题**: 整个数据层深度耦合 Supabase 生态

**影响范围**:

| 耦合点 | 文件 | 影响程度 |
|--------|------|---------|
| API 基地址 | `services/apiService.ts` | 全局 |
| 认证 Token | `utils/supabase/info.tsx` | 全局 |
| Auth 流程 | `contexts/AuthContext.tsx` | 认证系统 |
| Edge Functions | `supabase/functions/server/` | 后端 |
| KV 存储 | `supabase/functions/server/kv_store.tsx` | 数据持久化 |
| Token 刷新 | `apiService.ts` 内置刷新逻辑 | 会话管理 |

**具体表现**:

1. `utils/supabase/info.tsx` 硬编码 Supabase `projectId` 和 `publicAnonKey`
2. `apiService.ts` 中 `API_BASE` 直接拼接 Supabase Edge Functions URL
3. `AuthContext.tsx` 的登录/注册/刷新逻辑完全依赖 Supabase Auth API
4. 401 自动刷新逻辑绑定 Supabase JWT 生命周期
5. 后端 Hono Server 运行在 Deno Runtime (Supabase Edge Functions)

### 2.2 🟡 P1 — 缺少构建配置与包管理

**问题**: 根目录缺少标准前端工程化配置文件

| 缺失文件 | 影响 |
|---------|------|
| `package.json` | 无法安装依赖、运行脚本 |
| `tsconfig.json` | TypeScript 编译配置缺失 |
| `vite.config.ts` / `next.config.js` | 构建工具配置缺失 |
| `.env.example` | 环境变量模板缺失 |
| `.gitignore` | 版本控制忽略规则缺失 |
| `tailwind.config.ts` | Tailwind 自定义配置缺失 |

### 2.3 🟡 P1 — 状态管理策略待统一

**问题**: 当前使用 `useState` + `useMemo` + `localStorage` 的分散式状态管理

- 跨组件状态传递依赖 prop drilling
- 用户数据在 `AuthContext` 和各组件间重复计算
- 无全局状态管理方案 (如 Zustand / Jotai)

### 2.4 🟢 P2 — 代码组织优化空间

| 方面 | 现状 | 建议 |
|------|------|------|
| API 客户端 | 单文件 730 行 | 按域拆分 (auth/user/course/admin) |
| 类型定义 | 单文件 400+ 行 | 按域拆分 |
| Mock 数据 | 散布在多个文件 | 统一至 mock/ 目录 |
| 路由管理 | switch-case 硬编码 | 引入路由配置表 |

## 三、Supabase 替代方案

### 3.1 替代架构设计

```
┌────────────────────────────────────────────────────┐
│                   替代架构方案                        │
├────────────────────────────────────────────────────┤
│                                                    │
│  ┌──────────────┐    ┌──────────────┐             │
│  │  前端 React   │    │  API 网关     │             │
│  │  App.tsx      │───▶│  Hono/Express │             │
│  └──────────────┘    └──────┬───────┘             │
│                             │                      │
│              ┌──────────────┼──────────────┐       │
│              ▼              ▼              ▼       │
│  ┌──────────────┐ ┌──────────────┐ ┌────────────┐ │
│  │  认证服务     │ │  业务服务     │ │  AI 服务   │ │
│  │  JWT 自签发   │ │  REST API    │ │  多Provider│ │
│  └──────┬───────┘ └──────┬───────┘ └─────┬──────┘ │
│         │                │               │        │
│         ▼                ▼               ▼        │
│  ┌──────────────────────────────────────────┐     │
│  │            数据持久层                      │     │
│  │  SQLite (开发) / PostgreSQL (生产)         │     │
│  │  Prisma ORM / Drizzle ORM                 │     │
│  └──────────────────────────────────────────┘     │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 3.2 技术选型推荐

| 功能域 | 当前 (Supabase) | 推荐替代 | 理由 |
|--------|----------------|---------|------|
| 认证 | Supabase Auth | JWT + bcrypt 自实现 | 完全自主，无第三方依赖 |
| API 服务 | Supabase Edge Functions (Deno) | Node.js + Express/Hono | 标准 Node.js 生态 |
| 数据库 | Supabase PostgreSQL (KV 模拟) | SQLite (dev) / PostgreSQL (prod) | 真正的关系型数据库 |
| ORM | 无 (手写 SQL) | Prisma / Drizzle | 类型安全，迁移管理 |
| 文件存储 | Supabase Storage (预留) | 本地 / S3 兼容 | 按需选择 |
| 实时通信 | 无 | WebSocket (Socket.io) | AI 流式输出 + 通知 |

### 3.3 迁移路径

```
Phase 1: 解耦 Supabase 依赖
  ├── 创建 adapters/storage.ts (存储抽象层)
  ├── 创建 adapters/auth.ts (认证抽象层)
  └── 创建 adapters/api.ts (API 抽象层)

Phase 2: 实现本地后端
  ├── server/ 目录 (Node.js + Express)
  ├── db/ 目录 (Prisma schema + migrations)
  └── auth/ 目录 (JWT 签发/验证)

Phase 3: 数据迁移与验证
  ├── 数据导出 (Supabase KV → SQLite/PostgreSQL)
  ├── API 兼容性测试
  └── E2E 验证
```

## 六、结论

YYC³ 项目前端架构成熟度高，UI 层技术选型合理，类型系统完善。**核心瓶颈在于 Supabase 强耦合**，需要通过适配器模式进行解耦，替换为自主可控的 Node.js + SQLite/PostgreSQL 方案。

建议按 **Phase 1 (解耦) → Phase 2 (替换) → Phase 3 (优化)** 的渐进式路径推进，确保每个阶段都可独立交付和验证。
